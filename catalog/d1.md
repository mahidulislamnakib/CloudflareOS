# Cloudflare D1

Cloudflare D1 is Cloudflare's serverless SQL database for Workers applications.

Use D1 when your Cloudflare app needs structured data, tables, relationships, filtering, sorting, and queryable records.

---

## Simple explanation

D1 is a database you can query with SQL from Cloudflare Workers.

Think of it as the place where your app stores structured records such as articles, users, jobs, orders, settings, categories, and form submissions.

---

## What problem it solves

D1 solves the problem of storing relational application data without managing a separate database server.

It is useful when you need:

- tables
- records
- relationships
- search/filter queries
- sorting
- pagination
- migrations
- structured app data

---

## When to use D1

Use D1 when your app needs SQL-style structured data.

Good examples:

- CMS articles
- news portal content
- job posts
- user profiles
- form submissions
- product records
- categories and tags
- admin settings
- audit logs
- order records

---

## When not to use D1

D1 may not be the best fit when:

- you need to store large files
- you only need simple key-value config
- you need high-volume analytics events
- you need complex external database features
- you need a traditional database server outside Cloudflare

Use R2 for files.

Use KV for small key-value config/cache.

Use Analytics Engine for high-volume analytics-style events.

---

## Beginner example

A news portal might use D1 tables like:

```text
articles
authors
categories
tags
article_tags
media
settings
```

Basic flow:

```text
Worker route
  ↓
Validate request
  ↓
Query D1
  ↓
Return JSON or render page
```

## Production notes

For production projects:

- design the schema before writing routes
- use migrations instead of manual changes
- add indexes for common filters
- add pagination for list endpoints
- validate all write input
- avoid destructive schema changes without review
- keep file contents in R2, not D1
- backup or export important data
- add audit logs for admin actions

---

## Security notes

Do:

- validate all user input
- use prepared statements and bindings
- protect write routes with authentication
- limit public query size
- avoid storing secrets in D1
- avoid logging sensitive data
- use role checks for admin actions

Do not:

- expose admin tables publicly
- accept raw SQL from users
- store uploaded files directly in D1
- run migrations blindly in production
- skip backups for important data

---

## Cost awareness

D1 cost can be affected by:

- number of reads
- number of writes
- query volume
- inefficient queries
- missing indexes
- unnecessary repeated reads

Keep queries focused. Cache public read-heavy data with Cloudflare cache or KV when appropriate.

---

## Common mistakes

- using D1 for file storage
- forgetting migrations
- missing indexes on filtered columns
- building list APIs without pagination
- exposing write endpoints without auth
- changing schema without a rollback plan
- duplicating data that should be relational
- using KV when SQL relationships are needed

---

## Related Cloudflare services

| Service | Relationship |
| --- | --- |
| Workers | Runs code that queries D1 |
| R2 | Stores files linked from D1 records |
| KV | Stores small config/cache, not relational data |
| Queues | Handles async database-related jobs |
| Durable Objects | Coordinates state when needed |
| Pages | Can use backend routes that talk to D1 |
| Wrangler | Creates databases and runs migrations |
