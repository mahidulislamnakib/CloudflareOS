# Cloudflare R2

Cloudflare R2 is object storage for files, media, documents, backups, and other unstructured data.

Use R2 when your Cloudflare app needs to store uploaded files or generated assets without putting them inside the database.

---

## Simple explanation

R2 is where your app stores files.

D1 stores records. R2 stores objects.

For example, a news article record belongs in D1, but the article image belongs in R2.

---

## What problem it solves

R2 solves the problem of storing files without running your own storage server.

It is useful when you need:

- image uploads
- profile photos
- documents
- invoices
- receipts
- media files
- backups
- exports
- downloadable files
- generated assets

---

## When to use R2

Use R2 when your app needs file or object storage.

Good examples:

- article images for a news portal
- user portfolio files
- job attachments
- course thumbnails
- product photos
- delivery files
- admin media library
- PDF exports
- backup files

---

## When not to use R2

R2 may not be the best fit when:

- you need relational tables
- you need SQL queries
- you only need small config values
- you need high-frequency counters
- you need live shared state

Use D1 for structured records.

Use KV for small config/cache.

Use Durable Objects for coordinated live state.

---

## Beginner example

A CMS might store:

```text
D1:
articles table
media table

R2:
media/articles/2026/06/article-image.webp
```

Basic flow:

```text
User uploads file
  ↓
Worker validates file
  ↓
Worker stores file in R2
  ↓
Worker stores metadata in D1
  ↓
App uses R2 object key when needed
```

---

## Production notes

For production projects:

- validate file type
- validate file size
- generate safe object keys
- never trust original filenames
- store metadata in D1 if files need management
- separate public and private uploads
- plan cleanup for unused files
- avoid exposing sensitive objects publicly
- use signed access patterns when needed

---

## Security notes

Do:

- validate file type and size
- use safe generated object keys
- protect private files
- scan or review risky uploads when needed
- store only metadata in D1
- use role checks for admin uploads
- restrict public upload endpoints

Do not:

- store real secrets in object metadata
- trust user-provided filenames
- make all buckets public by default
- store uploaded files directly in D1
- allow unlimited file size
- expose private documents without checks

---

## Cost awareness

R2 cost can be affected by:

- storage volume
- request volume
- object count
- large media files
- repeated reads
- lifecycle and cleanup behavior

Use image optimization and cleanup rules when your project grows.

---

## Common mistakes

- using D1 for files instead of R2
- using one flat folder for every file
- not validating uploads
- not storing metadata when management is needed
- exposing private files publicly
- not planning object key structure
- forgetting cleanup for abandoned uploads
- assuming original filenames are safe
